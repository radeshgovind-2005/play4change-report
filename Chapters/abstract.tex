U!REKA Play4Change is a multiplatform gamified educational platform designed to promote sustained engagement with sustainability and digital education, developed as a final year undergraduate project at ISEL. The platform addresses two structural problems common to existing educational software: the manual content authoring bottleneck, where human effort must scale proportionally as the platform grows, and mobile platform fragmentation, where maintaining separate native applications for Android and iOS doubles engineering cost. Both problems are resolved simultaneously by the same architectural foundation.

The mobile client is built with KMP (Kotlin Multiplatform) and Compose Multiplatform, delivering a fully functional Android application and an iOS target from a single shared codebase. New educational tasks are produced automatically by a nightly batch pipeline that invokes the Mistral language model API through the LangChain4j library. Executing generation at a scheduled time rather than on user request eliminates inference latency from the user-facing request path and decouples platform availability from AI service availability entirely. Content repetition across days is prevented through semantic deduplication: each generated task is embedded as a high-dimensional numerical vector and compared against all previously stored vectors using cosine similarity within PostgreSQL extended with the pgvector module, discarding tasks whose meaning is too similar to existing content regardless of phrasing differences.

Player identity is managed through OAuth~2.0 delegated authentication combined with a dual-token session architecture: a short-lived access token held only in application memory and a long-lived refresh token rotated on every use, enabling automatic detection of stolen credentials. A server-side scoring engine assigns configurable rewards for timely correct completions and penalties for inactivity, sustaining the daily engagement loop. The backend runs on Spring Boot and is deployed as a multi-container application via Docker Compose. The batch cost model scales with the number of subject domains rather than user count, making the platform progressively more cost-efficient as adoption grows.
