## 1. What We Are Building and Why It Is Hard

### The Platform

U!REKA Play4Change is a multiplatform gamified educational platform. Players receive a new educational task every day, generated by an artificial intelligence, aligned with sustainability and digital education themes. Completing tasks earns points and badges. Inactivity triggers score penalties. The goal is sustained daily engagement with educational content.

### The Engineering Constraints

These are not business requirements. These are engineering constraints — properties the system must have for it to function correctly in the real world.

| Constraint | Why It Creates an Engineering Problem |
|---|---|
| Content is AI-generated | AI inference is slow, expensive, and non-deterministic. It cannot be naïvely placed on the user's request path. |
| Content is delivered daily | The system has a predictable daily rhythm. This creates specific caching and pre-computation opportunities — and specific failure windows. |
| The platform is mobile-first | Mobile networks are unreliable. The application must function correctly whether or not a connection is available. |
| Users are in the EU | Data residency and privacy regulations constrain where data can be processed and how identity is handled. |
| Player scores represent real effort | Incorrect scores — double-counting, lost updates, stale reads — are not minor bugs. They break the core contract of a gamified system. |
| A single developer maintains the system | Every architectural component must be independently debuggable, deployable, and replaceable. Complexity has a real operational cost. |

Every engineering decision in this document exists because of one or more of these constraints.

## 2. Generating AI Content — The Core Challenge

### The Problem

The platform's defining feature is AI-generated educational content. A player opens the app and receives a task. Somewhere, a language model must generate that task. The engineering question is not *whether* to use AI — it is **when** the AI runs and **who waits for it**.

### Why Timing Matters

Language model inference is not instant. Generating a task involves sending a request to an AI service, waiting for it to process the input, and receiving the output. This takes, depending on model size and server load, between two and eight seconds.

Two to eight seconds is unacceptable as a response time for a mobile application. A user who opens the app and sees a loading screen for five seconds before seeing content will stop opening the app.

Beyond latency: the AI service is a third party. It can be unavailable. It can be rate-limited. It can fail. If the system only generates content when a user requests it, then every user who opens the app during a third-party outage receives an error.

Beyond availability: the AI service charges per call. If every user triggers a generation on every session, cost scales directly with usage. A platform that grows cannot afford to have its primary infrastructure cost grow at the same rate.

### Candidate Approaches

**Candidate A: Synchronous on-demand generation**
Generate content at the moment the user requests it. The user waits while the AI produces the task.

*Consequences:*
- Request path latency of 2–8 seconds on every cold task fetch
- Platform availability directly coupled to AI service availability
- Cost scales linearly with active users
- No opportunity to inspect or filter content before delivery

**Candidate B: Asynchronous on-demand generation with polling**
Accept the request immediately, begin generation in the background, and have the client poll until content is ready.

*Consequences:*
- Request path is unblocked immediately
- Client complexity increases: must implement polling, handle timeout, show a waiting state
- Cost still scales with users
- Still coupled to AI service availability at the moment of user request
- Content still bypasses any quality gate

**Candidate C: Streaming generation**
Begin generating and send content to the client token by token as it is produced, so the user sees words appearing rather than waiting for the complete response.

*Consequences:*
- First content appears in approximately 300ms — perceived latency is much lower
- Platform still coupled to AI service at request time
- Cost still scales with users
- Partial content on screen creates inconsistent UI states
- Complex to implement reliably on mobile clients

**Candidate D: Scheduled batch pre-generation**
Run a background job at a fixed time each night, generate all content needed for the following day, store it, and serve it from storage when users arrive.

*Consequences:*
- Zero latency on user request path — content is already in storage
- AI service failures are entirely contained within the batch window — users are never affected
- Cost is fixed: proportional to number of subject domains and tasks per domain, not user count
- Content passes through quality gates before any user sees it
- Content is not dynamically personalised per user per session (acceptable for this platform's educational model)

### Trade-offs

| | Latency | Cost Model | Availability Dependency | Quality Gate |
|---|---|---|---|---|
| Synchronous | ❌ 2–8s | ❌ Per-user | ❌ Coupled | ❌ None |
| Async + polling | ✅ Non-blocking | ❌ Per-user | ❌ Coupled | ❌ None |
| Streaming | ✅ Perceived low | ❌ Per-user | ❌ Coupled | ❌ None |
| **Batch pre-generation** | ✅ **Zero** | ✅ **Fixed** | ✅ **Decoupled** | ✅ **Full** |

### Choice: Scheduled Batch Pre-Generation

The batch approach is the only candidate that solves all four problems simultaneously. The trade-off surrendered — real-time personalisation — is not a requirement of this platform. Tasks are domain-specific and level-appropriate; per-session personalisation is an extended objective that can be layered on without changing this fundamental architecture.

**The key engineering insight:** By separating *when content is generated* from *when content is delivered*, the system converts a runtime dependency (AI service must be up when the user is active) into a build-time dependency (AI service must be up at 23:00 each night). The failure surface shrinks from all hours to a single known window, and fallback strategies for that window are tractable.

**Cost model:**
```
Batch:      domains × tasks_per_domain × cost_per_task   →  fixed daily cost
On-demand:  active_users × cost_per_task                 →  grows with users
```
At 1,000 users, the batch model is roughly 10× cheaper. At 10,000 users, it is 100× cheaper. The cost advantage compounds with growth.

## 3. Ensuring AI Content Does Not Repeat

### The Problem

A language model has no memory between invocations. If you ask it to generate a sustainability task today, and ask again tomorrow, it has no recollection of what it generated previously. Over days and weeks, it will produce tasks that are conceptually equivalent — the same educational idea phrased differently.

A player who receives "sort your recycling correctly" on day 3 and "separate your waste into the right bins" on day 18 will notice. Not because the words are identical, but because the *meaning* is the same. They are being asked to do the same thing. The game feels repetitive. Engagement drops.

**The naive solution** — check if the exact text was generated before — fails entirely. Language models produce different phrasings of the same concept every time. Exact text matching catches nothing.

**The correct framing:** The platform needs to detect whether two pieces of text have the same *meaning*, not the same *words*.

### Candidates for Semantic Comparison

**Candidate A: Keyword overlap (Jaccard similarity)**
Count how many keywords two tasks share divided by the total number of unique keywords. Simple and fast.

*Consequence:* "Sort your recycling" and "Separate your waste into bins" share almost no keywords despite being semantically identical. "Reduce your electricity usage" and "Lower your power consumption at home" share no keywords at all. Keyword overlap is a measure of vocabulary similarity, not semantic similarity. This approach fails for paraphrase detection.

**Candidate B: Rule-based domain tagging**
Tag every task with a topic label ("recycling", "energy", "water") and reject new tasks that share a topic with recently used ones.

*Consequence:* Reduces repetition at the topic level but is too coarse. There are many distinct tasks within "recycling" and many distinct tasks within "energy". Blocking an entire topic to avoid repetition wastes the available content space. This approach over-blocks.

**Candidate C: Vector embeddings with similarity threshold**
Represent every piece of text as a high-dimensional numerical vector — an embedding — where texts with similar meanings produce vectors that point in similar directions. Measure the similarity between a new task's vector and all stored task vectors. Reject the new task if any similarity exceeds a defined threshold.

*Consequence:* Captures semantic equivalence regardless of phrasing. "Sort your recycling" and "Separate your waste into bins" produce vectors that are nearly identical in direction, even though they share no words. This is the correct tool for the problem. The cost is the need to store and query these vectors efficiently.

### Trade-offs

| Approach | Catches Paraphrases | Precision | Storage | Query Cost |
|---|---|---|---|---|
| Keyword overlap | ❌ | Low | Minimal | O(1) |
| Topic tagging | Partially | Too coarse | Minimal | O(1) |
| **Vector embeddings** | ✅ | **High** | Moderate | O(n) naïve / O(√n) indexed |

### Choice: Vector Embeddings with Approximate Search

Vector embeddings are the only approach that solves the actual problem. The additional storage and query cost are acceptable because this check only runs during the nightly batch — not on the user request path.

**The similarity metric — why angle, not distance:**
Two ways to measure how similar two vectors are:

*Euclidean distance* measures the straight-line distance between the tips of the vectors. If two sentences are semantically identical but one is longer than the other, their embedding vectors will have different magnitudes — different lengths. The Euclidean distance will be large even though the meaning is the same.

*Cosine similarity* measures the angle between the vectors, ignoring their length. Two semantically identical sentences, regardless of length, will produce vectors pointing in nearly the same direction. Cosine similarity correctly identifies them as equivalent.

For semantic comparison, the angle is what matters. Cosine similarity is the correct metric.

**The query efficiency problem:**
With thousands of stored tasks, comparing a new task's vector against every stored vector is O(n) — it gets slower linearly as the task library grows. An index that clusters vectors into groups and searches only the nearest groups reduces this to approximately O(√n). At the cost of occasionally missing the absolute nearest neighbour, it returns results fast enough to run inside a nightly batch without becoming a bottleneck.

For deduplication purposes, approximate is sufficient. Whether a task is 91% or 93% similar to an existing one is irrelevant — both would be correctly rejected.

## 4. AI Content Quality Before It Reaches Users

### The Problem

Language models sometimes produce malformed output. A task might be missing required fields. The response might not conform to the expected structure. The content might be incoherent. In a synchronous system, a malformed response can be retried immediately. In a batch system, malformed content must not silently enter the storage layer and later be served to users.

### Candidates

**Candidate A: Trust the model, validate nothing**
Accept all model output as-is. Simple.

*Consequence:* Malformed tasks reach users. Players see broken content. Trust in the platform is damaged. This is not a candidate for a production system.

**Candidate B: Validate structure at insert time**
Check that the response matches the required schema before writing to storage. Reject and discard if it does not.

*Consequence:* Malformed content never reaches storage. The batch job generates fewer tasks than intended on failure nights, but no bad content is delivered.

**Candidate C: Validate structure and retry on failure**
If validation fails, retry the generation with the same parameters. Allow a fixed number of retries before discarding.

*Consequence:* Maximises the number of valid tasks generated per batch run. Retries consume additional API calls (and cost), but only when validation fails — not as a default.

### Trade-offs

| Approach | User Impact | Batch Completeness | Cost |
|---|---|---|---|
| Trust the model | ❌ Malformed content reaches users | Best | Lowest |
| Validate, discard | ✅ No bad content | Moderate | Low |
| **Validate, retry** | ✅ **No bad content** | **Best** | Slightly higher |

### Choice: Validate Structure, Retry on Failure, Discard if Exhausted

The validation check runs synchronously inside the batch job before any write. If the model response does not conform to the required schema, the job retries up to a fixed number of times. If retries are exhausted, the task slot is discarded. Fewer tasks that day is preferable to one broken task delivered to a player.

The schema defines what a valid task is: required fields, field types, length constraints. Anything outside this definition is malformed and is not stored.

## 5. Choosing What to Persist and How to Model It

### The Problem

The platform stores multiple categories of data with fundamentally different characteristics:

- **Player identity and credentials** — must be accurate, must never be lost
- **Player scores and progress** — must be transactionally correct; partial updates are worse than no update
- **Task content and embeddings** — write once, read many; grows predictably over time
- **Semantic vectors** — high-dimensional floating point arrays requiring specialised query operations

The question is whether one storage system can serve all these needs, or whether specialised systems should be used for each.

### Candidates

**Candidate A: A single general-purpose relational database**
Store everything in one place. Relational databases offer ACID transaction guarantees across all operations. A single system simplifies operations, reduces failure modes, and eliminates synchronisation between systems.

*Consequence:* Must verify that the chosen database supports vector operations natively. Without this, semantic search requires either a full table scan (slow) or a second system.

**Candidate B: A relational database for structured data plus a dedicated vector database**
Use a relational database for player data and scores; use a separate, purpose-built vector database for embeddings and semantic search.

*Consequence:* Each system is optimised for its workload. Operational complexity doubles: two systems to deploy, monitor, back up, and keep in sync. If a task is written to the relational database but the write to the vector database fails, data is inconsistent. Synchronisation logic is non-trivial to make reliable.

**Candidate C: A document database for structured data plus a dedicated vector database**
Document databases offer schema flexibility but weaker transaction guarantees.

*Consequence:* Schema flexibility is not needed — the task schema is defined and stable. Weaker transaction guarantees introduce the risk of partial updates to score-critical data. The two-database operational burden remains.

### Trade-offs

| Approach | ACID Guarantees | Vector Search | Operational Complexity | Data Consistency |
|---|---|---|---|---|
| Single relational + vector extension | ✅ Full | ✅ Native | ✅ Low | ✅ Single transaction |
| Relational + dedicated vector DB | ✅ Full | ✅ Optimal | ❌ High | ⚠️ Cross-system sync |
| Document + dedicated vector DB | ⚠️ Weak | ✅ Optimal | ❌ High | ❌ Multiple weak systems |

### Choice: A Single Relational Database with a Vector Extension

A relational database with a native vector extension provides ACID guarantees for all data — including vectors — within a single transaction. A score update and the associated completion record write either both succeed or both fail, atomically. There is no synchronisation problem because there is one system.

**Why ACID matters specifically for this platform:**

*Atomicity:* When a player completes a task, two writes must happen together: the score increases, and the task is marked complete. If the process crashes between these two writes, the player either has points for a task they are not marked as having completed (exploitable) or is marked as having completed a task with no points (unfair). Atomicity guarantees neither partial state can persist.

*Isolation:* If a player submits two task completions rapidly — a double-tap on a mobile interface — both requests arrive at the server nearly simultaneously. Without isolation, both read the current score, both add to it, and one write silently overwrites the other. The player receives half the intended points. Isolation prevents this.

*Durability:* A confirmed score survives a server crash. The player earned those points. The commitment to the database is the commitment to the player.

**Managing growth of the task table:**

Tasks accumulate daily. Over months, the task table grows large. Without partitioning, every query scans the full table regardless of how old the data is.

The query pattern is almost always time-scoped: "give me today's tasks." Partitioning the table by time period means queries for recent tasks touch only the current partition. Historical data exists but is invisible to current queries. Old partitions can be archived without affecting active operations.

**The connection limit problem:**

Each connection to a relational database is a heavyweight resource — an operating system process consuming memory on the database server. Without management, a spike in application traffic creates a spike in database connections. The database has a hard limit; beyond it, new connections are refused.

The solution is a connection pool: a fixed set of pre-established connections shared across all application threads. Requests borrow a connection from the pool, use it, and return it. The pool is sized to the database's capacity. Traffic spikes queue briefly rather than overwhelming the database. The size is not arbitrary — it should be calculated from the database server's CPU capacity. More connections than the server can effectively parallelize reduce throughput rather than increasing it.

## 6. Keeping the Database Correct Under Concurrent Load

### The Problem

Multiple requests arrive at the server simultaneously. When multiple requests touch the same data — the same player's score, the same task completion record — they can interfere with each other in ways that produce incorrect results.

**The lost update problem:**
```
Current score: 100

Request A reads score: 100
Request B reads score: 100
Request A computes: 100 + 10 = 110, writes 110
Request B computes: 100 + 15 = 115, writes 115

Final score: 115
Expected score: 125
```

Request A's write was silently overwritten. Ten points were lost. This is a race condition — a class of bug where correctness depends on the relative timing of concurrent operations.

### Candidates

**Candidate A: Pessimistic locking**
Before reading a row, acquire an exclusive lock on it. No other transaction can read or write it until the lock is released. The second concurrent request waits until the first completes.

*Consequence:* Correctness is guaranteed. Throughput is reduced — every operation on the same player's score is serialised. Risk of deadlock if two transactions lock rows in different orders and each waits for the other.

**Candidate B: Optimistic locking with version numbers**
Add a version counter to every row. Read the row and note its version. When writing, assert that the version in the database is still the version you read. If another transaction changed it in the meantime, the write fails and the application retries.

```
Read:  score = 100, version = 7
Compute: new_score = 110
Write:  SET score = 110, version = 8 WHERE version = 7
        → succeeds if no one else changed version 7
        → returns 0 rows updated if version already changed → retry
```

*Consequence:* No locks held during processing. High throughput. Requires retry logic. Correct as long as conflicts (two requests for the same player at the same time) are infrequent.

**Candidate C: Atomic database operations**
Express the update as a single atomic increment: `score = score + 10` rather than read-compute-write.

*Consequence:* The database performs the read and write as a single atomic operation — no race window. No application-level retry needed. Limited to simple arithmetic operations; cannot handle complex conditional logic.

### Trade-offs

| Approach | Correctness | Throughput | Complexity | Risk |
|---|---|---|---|---|
| Pessimistic locking | ✅ Guaranteed | ❌ Serialised | Low | Deadlock |
| **Optimistic locking** | ✅ **Guaranteed** | ✅ **High** | Moderate | Retry overhead (rare) |
| Atomic operations | ✅ Guaranteed | ✅ High | Low | Limited expressiveness |

### Choice: Optimistic Locking for Complex Updates, Atomic Operations for Simple Increments

Score increments are expressed as atomic database operations where possible — simple, correct, fast. For updates involving multiple conditions or decisions (was this task already completed? should the penalty apply?), optimistic locking with a version counter handles the logic at the application layer with high throughput and guaranteed correctness.

**The TOCTOU problem for idempotency:**

A separate but related problem: a player completes a task and the network drops before the confirmation reaches the client. The client retries. Without protection, the score is applied twice.

The naive solution — check if the completion exists, then insert — has a race condition. Between the check and the insert, another request could have inserted the same completion. The check saw nothing; both insertions succeed.

The correct solution: a unique constraint at the database level on the combination of player and task. The database enforces this atomically during insert. The second insert fails with a constraint violation. The application catches this, recognises it as a duplicate rather than an error, and returns the same success response as the original. The player gets their points exactly once, regardless of how many times the client retries.

## 7. The Daily Nature of the Platform — Caching

### The Problem

U!REKA has a distinctive usage pattern: tasks are generated once per night and consumed throughout the following day. The same content is served to many users. This is the ideal scenario for caching — storing data close to where it is needed so it does not have to be fetched from the origin on every request.

Caching is a trade-off: it exchanges consistency (the cached value might not reflect the latest write) for speed and reduced load. The platform must decide what to cache, where to cache it, for how long, and what to do when the cache becomes stale.

### The Cache Hierarchy

There are multiple points in the system where data can be cached, each with different characteristics:

```
Player's Device    →    Application Server    →    Database    →    AI Service
(milliseconds)         (low milliseconds)         (5–50ms)          (2–8 seconds)

L1 cache               L2 cache                Source of truth     Generation origin
Fastest, smallest      Fast, shared            Authoritative       Slow, expensive
```

Each layer closer to the user is faster but holds less data and may be more stale. The art of caching is deciding which data belongs at which layer.

### The Core Trade-off: Staleness vs Speed

A cache is always potentially stale. The moment data is written to a cache, the source may be updated, and the cache becomes outdated. The longer data stays in the cache, the more likely it is to be stale. Reducing the cache lifetime improves freshness but reduces the benefit of caching. This tension is fundamental and cannot be eliminated — only managed.

### Candidates for Cache Placement: Task Content

**Candidate A: No caching — always fetch from database**
Every user request hits the database directly.
*Consequence:* Database receives every read request. At peak usage (students arriving in the morning), this creates a spike. No resilience against database unavailability.

**Candidate B: Server-side cache only**
The application server caches task content in memory. Clients always contact the server.
*Consequence:* Database is protected from repeated reads. Client still requires network to access content. No resilience against network failure.

**Candidate C: Client-side cache only**
Content is downloaded to the client and served locally on subsequent accesses. No server-side cache.
*Consequence:* Zero network requests for cached content. Device works offline after first load. Server still receives the initial fetch on every device on every day.

**Candidate D: Client-side cache with server-side backing cache**
Content is cached on both the server (protecting the database) and the client (enabling offline use). The client fetches from the server only if local cache is stale; the server fetches from the database only if its cache is stale.
*Consequence:* Multiple layers of protection. Database is shielded from traffic spikes. Clients work offline. Slightly more complexity in invalidation.

### Choice: Client-Side Cache with Server-Side Backing Cache

The layered approach is the correct choice because it solves two distinct problems with one design: it enables offline-first behaviour on the client, and it protects the database from repeated reads on the server.

**Cache invalidation strategy for task content:**

Task content is **immutable once generated**. A task written at 23:00 is not modified during the day. It is simply read.

For immutable data, the simplest invalidation strategy is correct: **time-based expiry at the boundary of the content's natural lifecycle**. Tasks expire at midnight, because that is when new tasks replace them. There is no need for complex event-driven invalidation — the content does not change mid-day, so the cache cannot become stale mid-day. Time-based expiry is not a simplification; it is the exact right tool for this data shape.

**Cache invalidation strategy for player scores:**

Player scores are the opposite of task content. They change frequently — every task completion, every penalty. A score that is stale by even one read cycle is incorrect.

For frequently-mutating data where consistency is required (see §9 for why scores are CP), the correct strategy is **write-through**: when a score is updated, the cache is updated synchronously as part of the same operation. The cache is never stale because it is never left behind. The cost is slightly slower writes — both the primary store and the cache must be written before success is confirmed. This is acceptable because score writes are infrequent compared to reads, and the consistency requirement demands it.

**The eviction policy:**

When a cache reaches its capacity limit, it must decide what to evict to make room for new entries. The choice of eviction policy determines which data stays and which is discarded.

*Least Recently Used (LRU):* Evict the entry that was accessed least recently. This keeps "hot" data — frequently accessed domains, popular badge definitions — in cache naturally. Rarely accessed entries fall out on their own. For a platform with many subject domains but where users tend to stay in their enrolled domains, LRU is the correct policy.

*Least Frequently Used (LFU):* Evict the entry with the fewest total accesses. Protects frequently accessed historical data even if it was not accessed recently. More complex to maintain. Not necessary for this platform's access pattern.

*First In, First Out (FIFO):* Evict the oldest entry regardless of access pattern. Simple but ignores actual usage — evicts hot data because it was loaded early.

**Choice: LRU.** The platform's access pattern is domain-centric. Students in the sustainability domain access sustainability content repeatedly. LRU keeps their content warm in cache. Domains no one is using fall out naturally, freeing space for active ones.

## 8. What Happens When the Network Fails

### The Problem

Mobile networks drop. Tunnels, basements, airplane mode, dead zones. A player who completes a task while the network is unavailable should not lose that completion. The platform must handle network unavailability gracefully — not by preventing interaction, but by deferring synchronisation.

### Candidates

**Candidate A: Block the action until connectivity is confirmed**
Do not allow task completion unless the network is available. Show an error if there is no connection.

*Consequence:* Simple. But it penalises players for circumstances outside their control. A player who completes a task in a tunnel cannot earn their points. This is a poor user experience and breaks the core engagement loop.

**Candidate B: Optimistic UI — assume success, reconcile later**
Mark the task as complete on the device immediately. Queue the score submission. Send it when connectivity returns. If the server rejects it (duplicate, validation failure), undo the local state.

*Consequence:* The player sees immediate feedback. Synchronisation happens in the background. Rollback on rejection is complex — the player may have already seen a score they do not actually have.

**Candidate C: Outbox pattern — queue locally, deliver reliably**
Before attempting a network operation, write the intent to a local persistent queue (the outbox). A background process continuously attempts to flush the outbox to the server. On server confirmation, the entry is removed from the outbox. If the app closes or the network drops, the outbox persists and is retried on next launch.

*Consequence:* The operation is guaranteed to eventually reach the server as long as the device eventually connects. The UI can show "pending" state until confirmation. Delivery is reliable. The server must be idempotent (§6) to handle the case where a queued submission is retried multiple times.

### Trade-offs

| Approach | User Experience | Reliability | Complexity |
|---|---|---|---|
| Block until connected | ❌ Poor | ✅ Simple | Low |
| Optimistic with rollback | ✅ Immediate | ⚠️ Rollback complexity | High |
| **Outbox pattern** | ✅ **Responsive** | ✅ **Guaranteed delivery** | Moderate |

### Choice: Outbox Pattern

The outbox pattern is the correct solution for reliable delivery in an unreliable network environment. Combined with server-side idempotency (duplicate submissions are harmless), it provides exactly-once semantics from the player's perspective: the completion is submitted once and applied once, regardless of how many times the client retried.

## 9. Consistency vs Availability — A Split Decision

### The Problem

In a distributed system, during a network disruption, you cannot have both perfect consistency (every reader sees the most recent write) and perfect availability (every request gets a response). This is not a limitation of implementation — it is a mathematical property of distributed systems known as the CAP theorem.

The practical consequence: every piece of data in the system must be classified. When a disruption occurs and a choice must be made, what matters more for this data — that the response is correct, or that a response is given at all?

### The Classification for U!REKA

This is not a single platform-wide decision. Different data has different requirements, and treating all data the same would mean either serving wrong scores (if we prioritise availability for everything) or showing empty screens (if we prioritise consistency for everything).

**Task content:**

*What staleness looks like:* A player receives yesterday's task because today's has not yet synced.  
*Impact:* The player has something to do. The content is real and valid — just not the freshest. The engagement loop continues.  
*What the alternative looks like:* A player opens the app and sees an error because the backend cannot confirm the latest content.  
*Impact:* The engagement loop breaks. A player who cannot engage today is less likely to return tomorrow.

**Classification: Availability over Consistency.** Stale content is better than no content.

**Player scores:**

*What staleness looks like:* A player sees 420 points when they actually have 430. Or worse: they complete a task and their points do not increase.  
*Impact:* The core contract of a gamified system is broken. Effort produces visible reward. If effort produces no visible reward, players stop making the effort. The platform fails at its fundamental purpose.  
*What the alternative looks like:* A score read that cannot be confirmed as current returns an error, and the player must retry.  
*Impact:* A brief wait is acceptable if it guarantees the score shown is accurate.

**Classification: Consistency over Availability.** An accurate score is more important than an always-available score.

### The Architectural Consequence

This split decision means the system cannot have a single unified storage and delivery strategy. Task content is served from a cache that may be slightly stale — this is correct. Score reads must come from the authoritative source — reads from a replica that may have replication lag are not acceptable for scores, even though they would be faster.

> **The engineering maturity here is not in knowing CAP. It is in recognising that a single platform can legitimately have different CAP requirements for different data, and designing the architecture to honour both simultaneously.**

## 10. Identity, Authentication, and Trust

### The Problem

The platform needs to know who each user is. Scores, progress, and badges belong to individuals. Without verified identity, anyone could submit completions on behalf of anyone else.

The naive solution — have users create accounts with passwords — introduces a new problem: the platform now holds sensitive credentials. If the database is breached, those credentials are exposed. Users often reuse passwords across services, so a breach of this platform compromises users' accounts elsewhere.

### Candidates

**Candidate A: Username and password**
Traditional account creation. Users provide a password; the platform hashes and stores it; it is compared on login.

*Consequence:* The platform holds credentials. A breach exposes hashed passwords, which can be cracked offline for weak passwords. Password reset flows create additional attack surface. The platform is now responsible for credential hygiene — a specialised and difficult security problem.

**Candidate B: One-time password via email or SMS**
User provides their email or phone. The platform sends a one-time code. The user enters the code to authenticate.

*Consequence:* No password stored. But the platform still manages a secret (the one-time code) and its delivery. Email delivery reliability is imperfect. If the user's email is compromised, their platform account is compromised.

**Candidate C: Delegated authentication via a trusted third party**
The user authenticates with a provider they already trust. That provider confirms the user's identity to the platform without the platform ever seeing a password. The platform only ever receives a token confirming identity.

*Consequence:* The platform never handles passwords. The credential storage attack surface is eliminated by design. Authentication is as strong as the third party's systems — which are hardened at a scale no individual application could match. The platform depends on the third party's availability, which in practice is very high.

### Trade-offs

| Approach | Credential Storage Risk | Implementation Burden | Dependency |
|---|---|---|---|
| Username + password | ❌ High | Moderate | None |
| One-time password | ⚠️ Low | High | Email/SMS delivery |
| **Delegated authentication** | ✅ **None** | **Low** | Third-party uptime |

### Choice: Delegated Authentication

Eliminating the credential storage problem entirely is the correct engineering decision. The attack surface is reduced by architectural choice, not by implementation effort. The dependency on the third party's availability is accepted because that availability is reliably high in practice.

After the third party confirms identity, the platform issues its own token — a signed, self-contained credential that the client presents on subsequent requests. This token is verified cryptographically without querying a database — any server instance can verify any token using the shared signing key. This is essential for the platform to scale horizontally without a shared session store.

## 11. Protecting Session Tokens After Issuance

### The Problem

Once a token is issued to a client, it can be stolen. A stolen token grants access until it expires. If the token is long-lived, the attacker has long-lived access. If the token is short-lived, the user must re-authenticate frequently, which is poor user experience.

This appears to be an irreconcilable tension: long-lived tokens are convenient but risky; short-lived tokens are safer but inconvenient.

### Candidates

**Candidate A: Single long-lived token**
Issue one token at login. It remains valid for days or weeks. Convenient. Stolen token provides long-term access.

*Consequence:* Acceptable user experience. Unacceptable security posture if tokens are stolen.

**Candidate B: Single short-lived token**
Issue one token at login valid for minutes. User must re-authenticate frequently. Stolen token is quickly useless.

*Consequence:* Tight exposure window. Frequent re-authentication interrupts the user experience — unacceptable for a daily engagement platform.

**Candidate C: Dual-token system with rotation**
Issue two tokens: a short-lived access token (valid for 15 minutes) and a long-lived refresh token (valid for 7 days). The access token is used on all API requests. When it expires, the refresh token is exchanged for a new access token — silently, without the user re-authenticating. When a refresh token is used, it is immediately invalidated and a new one is issued.

*Consequence:* The user never needs to log in again as long as they use the app within 7 days. The access token exposure window is 15 minutes. If a refresh token is stolen and used, the attacker receives a new token — but so does the victim the next time they use the app. The server detects that the old refresh token was presented again after already being rotated (replay detection). This is evidence of theft. The server invalidates all tokens for this user and forces re-authentication.

### Trade-offs

| Approach | User Experience | Exposure Window | Theft Detection |
|---|---|---|---|
| Single long-lived | ✅ Good | ❌ Days/weeks | ❌ None |
| Single short-lived | ❌ Frequent re-auth | ✅ Minutes | ❌ None |
| **Dual-token + rotation** | ✅ **Transparent** | ✅ **Minutes** | ✅ **Automatic** |

### Choice: Dual-Token System with Refresh Token Rotation

The dual-token architecture resolves the tension between convenience and security. The access token's short lifetime limits exposure. The refresh token's rotation detects theft. The user experiences neither repeated logins nor a security gap.

**Token storage on the client:**

The access token lives only in application memory. It is never written to persistent storage. If the application is closed, the token is gone — only the refresh token, stored in a secure, JavaScript-inaccessible location, persists. This means a successful cross-site scripting attack (where malicious code runs in the application context) cannot extract and exfiltrate a long-lived credential.

## 12. Scaling the Server Layer

### The Problem

Today the platform serves a student cohort. If it succeeds, it may serve thousands of students across partnered institutions. The question is how to design the server layer so that handling more load requires adding capacity, not rewriting software.

### The Statelessness Requirement

Scaling horizontally — adding more server instances and distributing load across them — only works if any instance can handle any request. If a server stores user-specific data in memory between requests (a session), then the next request from the same user must go to the same server. This creates dependencies between users and specific instances that break when instances are added, removed, or fail.

**The solution: stateless servers.** Every request must carry all the information the server needs to process it. The server processes the request and produces a response without relying on anything stored in memory from previous requests.

**How this is achieved:**
- Authentication tokens carry the user's identity in the request — no session lookup required
- All persistent state lives in the database — not in server memory
- Scheduled jobs use database-level locks to prevent two instances from executing the same job simultaneously

Any instance can handle any request. Adding instances requires no application code changes.

### Load Distribution

When multiple instances exist, incoming requests must be distributed across them. Several algorithms exist:

**Round robin:** Distribute requests in sequence across all instances. Simple, but ignores that requests have different weights. A request that triggers a database query and one that returns a cached value are not equivalent, but round robin treats them as equal.

**Least connections:** Send each request to the instance with the fewest currently open connections. Adapts to actual load rather than assuming equal request weight. Better than round robin for heterogeneous workloads.

**IP hash:** Route each client's requests to the same instance based on a hash of their IP address. Provides affinity — the same client always reaches the same server. Useful if instances have local state, but that would violate the statelessness requirement.

**Consistent hashing:** Useful for distributing data across nodes (cache partitions, for example) in a way that minimises remapping when nodes are added or removed. When a new node is added, only a fraction of keys need to move — those adjacent to the new node on the hash ring. Without consistent hashing, adding a node changes the assignment of nearly all keys, effectively invalidating the entire cache.

**Current decision:** Single instance — distribution is not yet active. The architecture is designed so that adding a reverse proxy in front with least-connections routing requires no changes to the application layer.

## 13. Cost as an Engineering Problem

### The Problem

Infrastructure and third-party services cost money. A system whose cost scales with its success is a system that penalises growth. Cost is not a business consideration appended to engineering decisions — it is an engineering constraint that shapes architectural choices.

### The Two Cost Models

**Linear cost model:** Cost grows proportionally with users. Every active user generates a certain number of expensive operations. As users grow, cost grows at the same rate.

*The problem:* A platform that becomes popular becomes expensive proportionally. Growth must be financed. The unit economics never improve.

**Fixed cost model:** Cost is determined by system parameters (number of domains, tasks per domain), not by user count. As users grow, cost stays flat.

*The consequence:* Beyond a certain user count, the platform becomes progressively more cost-efficient. Growth improves unit economics.

### The AI Generation Trade-off Quantified

The choice between on-demand and batch generation (§2) is also a cost choice:

```
On-demand model:
  Cost per day = active_users × cost_per_generation
  At 1,000 users:  1,000 × cost  (baseline)
  At 10,000 users: 10,000 × cost
  At 100,000 users: 100,000 × cost

Batch model:
  Cost per day = domains × tasks_per_domain × cost_per_generation
  At 1,000 users:  5 × 20 × cost  (approximately 10× cheaper than baseline)
  At 10,000 users: 5 × 20 × cost  (same — 100× cheaper than on-demand)
  At 100,000 users: 5 × 20 × cost (same — 1,000× cheaper than on-demand)
```

The batch architecture is not just a latency and availability choice. It is the difference between a cost model that collapses under success and one that improves with it.

### Infrastructure Sizing

Over-provisioning infrastructure is a cost. Under-provisioning causes failures. The correct approach is to size infrastructure to the actual load, with headroom for peaks, and a clear path to scale when thresholds are approached.

For a platform at initial deployment:
- A single server running all components is the appropriate starting point
- The stateless server design means horizontal scaling — adding instances — requires no code changes when needed
- Adding managed database infrastructure (replication, automated backups) is the natural next step as load grows

Choosing a distributed deployment model before it is needed adds operational complexity, infrastructure cost, and debugging difficulty with no corresponding benefit. This is not premature optimisation avoidance — it is engineering discipline.

## 14. Reliability — Designing for Failure

### The Problem

Every component of the system will eventually fail. Hardware fails. Networks drop. Third-party services have outages. The database runs out of disk space. The batch job crashes halfway through. A reliable system is not one that never fails — no such system exists. A reliable system is one that **fails gracefully**, **contains failures**, and **recovers automatically**.

### Availability Arithmetic

Availability is measured as the fraction of time the system is correctly serving requests.

```
Three components, each at 99% availability:
System availability = 0.99 × 0.99 × 0.99 = 97.03%
Annual downtime: 263 hours
```

Each dependency added to the critical path multiplies the risk. The goal is to minimise the critical path — the set of components that must all be functioning for any user-facing operation to succeed.

**The batch architecture's contribution to reliability:** The AI service is removed from the user-facing critical path entirely. A user fetching today's task does not depend on the AI service being available. The AI service is only critical during the nightly batch window — a known, manageable period.

### Failure Mode Analysis

**Batch job failure:**

*Cause:* AI service unavailable. Network timeout. Server out of memory.  
*Naive behaviour:* No tasks generated. Tomorrow's users see empty content.  
*Correct behaviour:*
- Retry with increasing delays between attempts (exponential backoff). This prevents hammering a rate-limited or recovering service, and gives it time to stabilise before the next attempt.
- If all retries are exhausted, fall back to a pre-authored static content library. Users receive content — not the best content, but content. The engagement loop continues.
- Alert the developer by 06:00 so the issue can be investigated before users are active.

**The idempotency requirement for retried jobs:** If a batch job crashes halfway through and is restarted, it must not create duplicate content. The semantic deduplication layer (§3) serves this purpose — any task generated in the first partial run is rejected as a duplicate on the second run. The job can be rerun safely.

**Database connection exhaustion:**

*Cause:* Traffic spike creates more concurrent requests than the connection pool can serve.  
*Naive behaviour:* Requests queue indefinitely. Memory fills with waiting requests. Server becomes unresponsive.  
*Correct behaviour:* The connection pool is configured to fail fast — after a short timeout, a request that cannot acquire a connection receives an error rather than queuing indefinitely. Fast failure is preferable to slow degradation: the client can retry; the server preserves stability.

**Score submission during network loss:**

*Handled by the outbox pattern (§8).* The completion is queued locally. It reaches the server when connectivity returns. Server-side idempotency prevents duplicate application.

### Graceful Degradation

Graceful degradation means the system loses capabilities in response to failures rather than failing completely. A hierarchy of degradation modes:

1. **Full functionality:** All services up, fresh content served, real-time score updates
2. **Degraded content:** Batch job failed, static fallback content served, game playable
3. **Offline mode:** No network, local cache served, completions queued locally
4. **Read-only mode:** Database write issues, scores shown from cache, submissions queued

The system never shows a completely broken state as long as local cache has been populated at least once. A player who has used the app before can always play — they may not get the most current content or see their score update in real time, but the core experience is preserved.

## 15. Knowing What Is Happening in Production

### The Problem

A system running in production is opaque by default. Users experience problems they cannot articulate. "It doesn't work" is not actionable. Without instrumentation, the developer has no way to know which component failed, how often, with what impact, or why.

Debugging a production system without observability is forensics — piecing together what happened after the fact. Debugging with observability is monitoring — seeing what is happening as it happens.

### The Three Questions Observability Answers

**Is the system healthy right now?** → Metrics  
Numbers that aggregate system behaviour over time: requests per second, error rate, latency distribution, tasks generated per night. These answer "is something wrong?" with quantitative evidence.

**What happened and in what order?** → Logs  
Timestamped records of events: this request arrived, this query ran, this generation failed, this completion was recorded. These answer "what exactly occurred?" with a sequential audit trail.

**Where did this specific request spend its time?** → Traces  
Records that follow a single request through all system components, showing how much time each component took. These answer "why was this request slow?" by pinpointing which component was the bottleneck.

### Why Aggregated Metrics Lie

The average latency of 100ms might mean all requests take approximately 100ms (good) or that 95% take 50ms and 5% take 1,050ms (bad — those 5% of users have a terrible experience). Averages obscure distribution.

**Percentile metrics** are the correct alternative:
- P50 (median): half of all requests are faster than this
- P95: 95% of requests are faster than this. The P95 user has a poor experience if this number is high.
- P99: 99% of requests are faster than this. The P99 user represents a tail case — but at 1,000 users, that is 10 real people per request batch.

Measuring and alerting on P95 and P99 latency ensures tail behaviour is visible and addressable.

### Why Structured Logs Outperform Plain Text

A log line like:
```
2026-03-09 23:01:42 INFO Generated task 4821 in 1840ms
```

Cannot be queried programmatically without parsing. "How many tasks took more than 2 seconds?" requires regex.

A structured log line:
```json
{"timestamp": "2026-03-09T23:01:42Z", "event": "task_generated",
 "task_id": 4821, "duration_ms": 1840, "domain": "sustainability",
 "tokens_used": 312, "trace_id": "a3f2c1d0"}
```

Can be filtered and aggregated directly: filter `event = "task_generated"`, sum `tokens_used` by `domain`, find all entries where `duration_ms > 2000`. No parsing required.

The `trace_id` field ties together every log line produced by a single operation. To reconstruct exactly what happened during a specific batch run or a specific user request, filter by trace ID. The complete sequence appears in chronological order.

## 16. Preventing Abuse and Protecting Integrity

### The Problem

A gamified system's integrity depends on scores reflecting genuine engagement. If a player can submit task completions programmatically — faster than any human could complete tasks — the leaderboard is meaningless and the educational purpose is defeated.

Beyond game integrity: without rate limiting, a single automated client can exhaust server resources, denying service to legitimate users.

### Candidates for Rate Limiting

**Fixed window counter:** Count requests in a fixed time window (e.g., per minute). Reset at the window boundary.

*Consequence:* Simple. But exploitable: a burst of requests at the end of one window and the start of the next — at the window boundary — produces a burst of `2N` requests in a short period while satisfying the per-window constraint.

**Sliding window:** Count requests in a rolling window that slides with time. No boundary burst vulnerability.

*Consequence:* More accurate. Requires storing timestamps of recent requests. Higher memory cost.

**Leaky bucket:** Requests enter a queue (the bucket) and are processed at a constant rate. If the bucket is full, new requests are rejected.

*Consequence:* Enforces a strictly constant output rate. Legitimate burst behaviour — a player catching up on three tasks they missed — is queued and delayed, even though it is not abuse. The constant rate is often too rigid for real usage patterns.

**Token bucket:** Each user has a bucket with a capacity of N tokens. Each request consumes a token. Tokens are replenished at a constant rate. If the bucket is empty, the request is rejected. Burst is allowed up to the bucket capacity; sustained high volume is blocked.

*Consequence:* Legitimate burst (completing a backlog of tasks quickly) is permitted up to the capacity. Sustained automated submission is blocked. This matches real human usage patterns.

### Trade-offs

| Algorithm | Burst Handling | Boundary Vulnerability | Complexity |
|---|---|---|---|
| Fixed window | ❌ Boundary burst | ❌ Yes | Low |
| Sliding window | ✅ Accurate | ✅ None | Moderate |
| Leaky bucket | ❌ Queues legitimate bursts | ✅ None | Moderate |
| **Token bucket** | ✅ **Allows legitimate burst** | ✅ **None** | **Moderate** |

### Choice: Token Bucket

Token bucket correctly models the distinction between legitimate burst (allowed, up to bucket capacity) and sustained abuse (blocked, because the refill rate limits it). This is the right algorithm when the goal is to allow human behaviour patterns while preventing automated abuse.

## 17. Delivering Data Efficiently to Clients

### The Problem

List endpoints — task history, badge catalogue, leaderboard — potentially return large datasets. Returning the entire dataset on every request is expensive (database scans the full table) and slow (large payloads over mobile networks). Pagination is required. But pagination has more than one design, and the designs have meaningfully different performance characteristics.

### Candidates

**Offset pagination:**
The client requests a page by specifying how many records to skip: "give me 20 records, starting from position 100."

*Consequence:* The database must scan through 100 records to discard them, then return 20. At page 1,000 (position 20,000), the database scans 20,000 records to return 20. Performance degrades linearly with page number. Additionally, if new records are inserted between page requests, records shift — page 2 may repeat records already seen on page 1, or skip records entirely.

**Cursor-based pagination:**
The client receives an opaque cursor representing its position in the dataset. To fetch the next page, it sends the cursor. The database translates this into "give me 20 records where the ID is greater than the last ID you returned."

*Consequence:* The database uses the primary key index directly. Page 1,000 is as fast as page 1 — both require only an index lookup. New insertions do not affect page boundaries, because pages are anchored to stable identifiers, not absolute positions.

### Trade-offs

| Approach | Performance at Depth | Consistency on Insert | Client Complexity |
|---|---|---|---|
| Offset | ❌ O(n) — degrades with depth | ❌ Pages can shift | Low |
| **Cursor** | ✅ **O(1) — constant** | ✅ **Stable** | Moderate |

### Choice: Cursor-Based Pagination

Cursor pagination is strictly superior for any dataset that grows. The moderate additional client complexity (storing and sending a cursor) is a fixed one-time cost. The performance improvement compounds with dataset size.

## 18. Reproducibility and Deployment Safety

### The Problem

Software that runs on a developer's machine may not run identically on a production server. Different operating system versions, different dependency versions, different environment variables — any difference can cause behaviour that works in development to fail in production. This is not a hypothesis. It is a routine engineering experience, captured in the phrase: "it works on my machine."

Beyond reproducibility: deploying a new version of the application must not require manual configuration steps that are easy to miss or get wrong. Manual steps are human error surfaces.

### Candidates

**Manual deployment — copy files to server, configure by hand**
Simple to describe. Impossible to reproduce reliably. Configuration state exists only in the memory of the person who set it up. Any deviation from the expected configuration is a silent bug until it manifests as a production failure.

**Environment-specific configuration management — scripts that install and configure dependencies on the target server**
More reproducible than manual. Still depends on the state of the server — what version of the runtime is installed, what other software is running, what configuration files exist from previous deployments.

**Containerisation — package the application and all its dependencies into a portable, isolated unit**
The application and everything it needs to run are captured together. The unit runs identically on any machine that can execute containers. Development environment, test environment, and production environment are identical by construction.

*Consequence:* "Works on my machine" and "works in production" are the same statement. Deployment is running a pre-built artefact, not re-constructing an environment. Rolling back to a previous version is replacing the running artefact with a previous one.

### Choice: Containerisation

Containerisation eliminates environment differences as a failure mode. This is not a convenience — it is an engineering control that prevents an entire class of production bugs.

**Secret management:**
Configuration values that grant access — signing keys, database passwords, external API credentials — must never appear in the application code, the container image, or version control history. If they do, they are effectively public. Once a secret appears in version control history, it must be considered compromised regardless of subsequent deletion, because history is retained.

Secrets are injected at runtime from outside the container — from the hosting environment's secret management facility. The container image contains the application, not its secrets. The application cannot run without secrets, but the secrets are not inside it.

**Health checks before traffic:**
An application container that starts does not mean the application is ready to receive traffic. The database may still be initialising. Configuration may still be loading. A health check — a lightweight endpoint that returns success only when the application is genuinely ready — prevents traffic from being routed to an instance before it can handle it. Without a health check, a restarting instance may receive requests during startup that it cannot process.
